1. HTTP 与 HTTPS 有哪些区别？

  • HTTP 是超文本传输协议，信息是 明文 传输，存在安全风险的问题。HTTPS 则解决 HTTP 不安全的缺陷，在 TCP 和 HTTP 网络层之间加入了 SSL/TLS 安全协议，使得报文能够 加密 传输。

  • HTTP 连接建立相对简单， TCP 三次握手之后便可进行 HTTP 的报文传输。而 HTTPS 在 TCP 三次握手之后，还需进行 SSL/TLS 的握手过程，才可进入加密报文传输。

  • HTTP 的端口号是 80，HTTPS 的端口号是 443。

  • HTTPS 协议需要向 CA（证书权威机构）申请数字证书，来保证服务器的身份是可信的。

  • HTTPS SEO 优先，HTTP 无优待

  • HTTPS 浏览器标识 安全，HTTP 相反

HTTPS 的缺点

  • 通信两端都需要进行加密解密，消耗大量 CPU、内存等资源，从而增加服务器负载；
  • 加密运算和多次握手降低了访问速度；
  • 因为信息都被加密了，开发阶段会加大调试难度；
  • 用 HTTPS 访问的页面，页面内的外部资源都要用 HTTPS 请求。

HTTP 的缺点

  • 明文传递，容易被窃听
  • 报文有被篡改的风险
  • 请求或响应有被伪造的风险。

2. HTTPS 是如何加密的

https 协议 = 包裹了 SSL 的 HTTP

http + 加密 + 认证 + 完整性保护 = https

https 是采用非对称加密 + 对称加密相结合的方式来进行加密的，在 HTTP/2 中 TCP 三次握手后会进入 SSL/TSL 握手，当 SSL/TSL 建立链接后，才会进行报文的传输。SSL/TLS 协议流程如下：

服务器拥有 1 个公钥 A 和 1 个私钥 B

  • （1）浏览器向服务器发起请求，请求包括
    • SSL/TSL 版本号
    • 加密套件，即客户端支持的加密算法列表
    • 生成第 1 个随机数（随机数 1）
    • 有一个 Client Hello 字符串
  • （2）服务器收到请求后，向客户端发出响应：
    • 确认 SSL/TSL 版本号，若客户端不支持，则关闭通信
    • 确认加密算法列表
    • 生成第 2 个随机数（随机数 2）
  • （3）客户端向服务器索要并验证服务器的公钥，服务器再次向客户端发送
    • 数字证书。服务器会将公钥注册到 CA（第三方证书机构），然后 CA 拿自己的私钥对服务器的公钥进行处理并颁发数字证书。
    • 公钥 A
    • Hello Done（表示发送完毕）
  • （4）客户端收到一系列响应后，确认数字证书和公钥 A，没问题后向服务端发送：
    • 生成 1 个随机数（随机数 3，也是 预主秘钥 X），此预主秘钥会通过公钥 A 进行加密，传输到服务器
    • 客户端握手结束通知，表示客户端握手结束。
  • （5）服务端收到客户端数据后，使用私钥 B 对加密后的预主秘钥 X 进行解密（没有其他人知道预主秘钥，因为它加密了，除非服务器私钥泄露）然后服务端通过第一、二、预主秘钥计算出 会话秘钥。客户端也计算出了会话秘钥。
  • （6）服务端向客户端发送
    • 加密通信算法改变通知，以后使用会话秘钥通信
    • 服务端握手结束
      到此为止，SSL/TSL 握手结束，在此之后都会通过 会话秘钥 来进行加密解密，也就是对称加密。

为什么需要加密？

客户端和服务器通信的过程中，如果内容不加密，有可能被通信过程中的任意一个节点（如代理服务器、路由器、wifi 热点、通信服务运营商等）获取并篡改。篡改者可以做到不被客户端和服务器双方察觉，即 中间人攻击。

对称加密

客户端和服务器约定好 秘钥，任何内容使用秘钥加密后进行传输，但秘钥容易被第三方得知并破解。

非对称加密

客户端和服务器都有自己的 公钥 + 私钥，使用公钥加密，私钥破解（或私钥加密，公钥破解）。但加密解密过程更复杂，更慢了。且公钥依旧不安全。

数字证书

客户端先向服务器端索要公钥，然后用公钥加密信息，服务器收到密文后，用自己的私钥解密。

这就存在些问题，如何保证公钥不被篡改和信任度？

所以这里就需要借助第三方权威机构 CA（数字证书认证机构），将服务器公钥放在数字证书（由数字证书认证机构颁发）中，只要证书是可信的，公钥就是可信的。通过数字证书的方式保证服务器公钥的身份，解决冒充的风险。

3. 在浏览器中，一个页面从输入 URL 到加载完成，都有哪些步骤？

（1）域名解析，根据域名找到服务器的 IP 地址
（2）建立 TCP 连接，浏览器与服务器经过 3 次握手后建立连接
（3）浏览器发起 HTTP 请求，获取想要的资源
（4）服务器响应 HTTP 请求，返回指定资源
（5）浏览器渲染页面，解析接收到的 HTML、CSS 和 JavaScript 等文件

关于浏览器渲染过程

  • 解析 HTML，生成 DOM 树，解析 CSS，生成 CSSOM 树
  • 将 DOM 树与 CSSOM 树结合，生成渲染树（Render Tree）
  • Layout（回流）：根据生成的渲染树进行回流，得到节点的几何信息（位置、大小）
  • Painting（重绘）：根据渲染树以及回流得到的几何信息，得到节点的绝对像素
  • Display：将像素发送给 GPU，展示在页面上

4. 对比持续通信的方法

（1）轮询

  • 通过 setInterval 或 setTimeout 定时获取并刷新页面上的数据
  • 定时轮询，不一定有新数据
  • 并发较多时，增加服务器负担

*（2）长连接

Connection: keep-alive

  • 减少无用请求
  • 并发较多时，增加服务器负担

*（3）长轮询

  • 服务端收到请求后，hold 住连接，直到新消息返回时才响应
  • 减少无用请求
  • 返回数据顺序无保证

（4）Web Sockets

  • 在客户端和服务器间打开交互式通信会话
  • 兼容 HTTP 协议。与 HTTP 同属应用层，默认端口是 80 和 443
    • 未加密连接：ws://（对应 http:// 端口 80
    • 加密连接：wss://（对应 https:// 端口 443
  • 数据格式轻量（支持二进制数据以及纯文本数据，传输过程中可能需要用到 JSON.stringify 和 JSON.parse），性能开销小，通信高效
  • 没有同源限制

JS 中创建 Web Socket 之后，会有一个 HTTP 请求发送到浏览器以发起连接。取得服务器响应后，建立的连接会从 HTTP 协议交换为 Web Socket 协议，也就是说，使用标准 HTTP 服务器无法实现 Web Sockets，只有支持这种协议的专门服务器才能正常工作。

// 创建一个 WebSocket 实例，并传入 URL
var socket = new WebSocket("ws://www.example.com/server.php")
// 注意，必须给 WebSocket 构造函数传入绝对 URL。同源策略对 Web Sockets 不适用，因此可以通过它打开到任何站点的连接。至于是否会与某个域中的页面通信，则完全取决于服务器。
var message = {
  time: Date.now(),
  text: "Hello world!"
}
socket.send(JSON.stringify(message));
// Web Sockets 只能通过连接发送纯文本数据，所以对于复杂的数据结构，在通过连接发送之前，必须进行序列化。

// 当服务器向客户端发来消息时，WebSocket 对象就会触发 message 事件。这个 message 事件与其他传递消息的协议类似，也是把返回的数据保存在 event.data 属性中。
socket.onmessage = function(event){
 var data = '';
 try {
   data = JSON.parse(event.data) // event.data 中返回的数据也是字符串。
 } catch {
   // PASS
 }
 //处理数据
};
socket.close()

WebSocket 对象的事件

事件 触发阶段
open 成功建立连接时
message 服务器向客户端发来消息时
error 发生错误时触发，连接中断
close 连接关闭时

socket.io 是个将 WebSocket 做了封装的跨平台 WebSocket 库，API 前后端一致，可以实现断线重连，丢包重发。可以触发和响应自定义事件，虽然还没看，但之后用到的话可以了解一下

5. 什么是 TCP 三次握手，为什么要三次握手

（1）什么是 TCP 三次握手

  • 起初
    • 客户端 CLOSED
    • 服务端 CLOSED
  • 第一次握手
    • 客户端发送携带 SYN 标志位的包，请求建立连接
  • 第二次握手
    • 服务器接收请求报文
      • 同意连接
        • 服务端响应一个携带 SYN 和 ACK 标志位的包
      • 拒绝连接
        • 服务端 REFUSED
  • 第三次握手
    • 客户端接收应答报文
      • 客户端发送一个携带 ACK 标志位的包，表示连接成功
      • 客户端 ESTABLISHED
    • 服务端接收确认报文
      • 服务端 ESTABLISHED

（2）为什么要三次握手

&emsp;&emsp;“已失效的连接请求报文段”的产生在这样一种情况下：client 发出的第一个连接请求报文段并没有丢失，而是在某个网络结点长时间的滞留了，以致延误到连接释放以后的某个时间才到达 server。本来这是一个早已失效的报文段。但 server 收到此失效的连接请求报文段后，就误认为是 client 再次发出的一个新的连接请求。于是就向 client 发出确认报文段，同意建立连接。假设不采用“三次握手”，那么只要server发出确认，新的连接就建立了。由于现在 client 并没有发出建立连接的请求，因此不会理睬 server 的确认，也不会向 server 发送数据。但server却以为新的运输连接已经建立，并一直等待 client 发来数据。这样，server 的很多资源就白白浪费掉了。采用“三次握手”的办法可以防止上述现象发生。例如刚才那种情况，client 不会向 server 的确认发出确认。server 由于收不到确认，就知道 client 并没有要求建立连接。

&emsp;&emsp;三次握手不是 TCP 本身的要求, 而是为了满足”在不可靠信道上可靠地传输信息”这一需求所导致的，三次通信是理论上的最小值。请注意这里的本质需求,信道不可靠, 数据传输要可靠. 当然后面还能继续握手，只是没必要了。

6. TCP 4 次挥手的过程

当要断开 TCP 连接时，通信两端就会进行 4 次挥手的操作。由于连接是双向的，所以客户端和服务端都要发送携带 FIN 标志位的包，才算彻底断开了连接，具体过程如下所列。

  • （1）客户端发送一个携带 FIN 标志位的包，请求断开连接；
  • （2）服务器响应一个携带 ACK 标志位的包，同意客户端断开连接；
  • （3）服务器再发送一个携带 FIN 标志位的包，请求断开连接；
  • （4）客户端最后发送一个携带 ACK 标志位的包，同意服务器断开连接。

7. HTTP 请求方法和状态码

请求方法

请求方法 语义 安全性 数据传递
GET 获取数据 更安全，因为 GET 只读取，不改变服务器中的数据，但参数可见，不适合密码等重要信息的传输 参数附带在 URL 上（params）可以用 decodeURIComponent(window.location.search) 获取
POST 提交数据 - 提交数据放在报文里（body）

因为浏览器中，URL 长度会被限制，所以 GET 请求能传递的参数长度是有限的（并不是 HTTP 的限制）

状态码

状态码 类别 原因短语
1XX 信息 请求已接受，正在处理中
2XX 成功 请求已成功处理
3XX 重定向 客户端需要附加操作才能完成请求
4XX 客户端错误 客户端发起的请求服务无法处理
5XX 服务器错误 服务器在处理请求时发生错误或异常

404（Not Found 未找到，无原因）500（Internal Server Error 服务端报错，通常是脚本报错）
