1. JSBridge 的通信原理

JSBridge 是 Hybrid App 中 JavaScript 与原生代码（iOS/Android）双向通信 的桥梁。其核心原理可以概括为：JS 发起调用 → 原生接收并处理 → 原生回调 JS。下面从两个方向和底层实现分别说明。

使用场景：App 内嵌 H5 页面需要调用原生能力（拍照、定位、分享、支付），或者原生需要通知 H5 页面刷新数据（如登录状态变化）。微信 JS-SDK、支付宝 JSAPI 底层都是 JSBridge。

JSBridge 是 Web 和 Native 之间的通信桥梁。

1. JS 调用 Native
“JS 无法直接调用系统 API，所以需要通过 WebView 提供的拦截机制。常用有两种方式：

  • URL Scheme 拦截：JS 发起一个自定义协议的请求（比如 myapp://getUser?params=xxx），原生拦截后解析执行。缺点是 URL 长度有限制，连续调用容易丢失。

  • 注入全局对象（更主流）：原生在 WebView 的 JS 上下文里挂一个对象，比如 window.NativeBridge，这个对象的方法直接由原生实现。JS 调用 window.NativeBridge.getUser(params, callback)，原生处理完成后，再通过执行 JS 回调把结果传回来。

实际项目中我们通常用第二种，因为更高效、灵活。

2. Native 调用 JS

原生主动给 JS 发消息，比如定位结果、支付结果、推送通知。实现很简单：原生直接执行 WebView 里的 JS 代码，比如 evaluateJavascript 或 loadUrl，调用我们前端预先挂载到 window 上的回调函数。
典型做法是：前端在调用原生时生成一个唯一 callbackId，原生处理完拿着这个 id 和结果，通过 window._handleCallback(id, result) 触发对应的回调。

如何处理 Bridge 还没加载？
用 window.addEventListener('bridgeReady', ...) 或者主动轮询检测 window.NativeBridge，未就绪时 push 到队列，就绪后批量执行。

怎么调试 JSBridge？
移动端用 vConsole 或 Chrome 远程调试；原生端打日志；也可以用 alert 或 prompt 模拟但会干扰。
