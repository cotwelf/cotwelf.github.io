1. React 中使用 useEffect 如何清除副作用

在 useEffect 中返回一个函数

// 例如我们在组件销毁时清除 setTimeout
const Component = () => {
  const [name, setName] = useState("");
  let timer = null;
  useEffect(() => {
    timer = setTimeout(() => {
      setName("cotwelf");
    }, 1000);
    return () => {
      clearTimeout(timer);
      timer = null;
      console.log("cleared");
    };
  }, []);
  return <div>{name}</div>;
};

2. 为什么 React useCallback 能减少渲染？

重点总结

useCallback 通过缓存函数引用，使其在依赖不变时保持引用稳定，从而在配合 React.memo 时避免子组件因 props 引用变化导致的无意义重新渲染。

useCallback 本身不会“减少渲染”，它只是“减少函数引用变化”，从而在特定条件下减少子组件的重新渲染。

在 React 中，父组件重新渲染时，内部定义的函数会重新创建（新内存地址）。

useCallback 做了什么？

const memoFn = useCallback(() => {
  doSomething();
}, [deps]);

本质是：

在依赖不变的情况下，返回同一个函数引用

也就是说：

每次组件 re-render

普通函数：都会重新创建（新引用）

useCallback：依赖没变 → 还是同一个引用

为什么“函数引用变化”会导致渲染？

React 的组件 props 比较是 浅比较（reference compare）

<Child onClick={fn} />

如果：

fn 是新函数（引用变了）

React 会认为 props 变了

即使逻辑没变，也可能触发 Child re-render

useCallback 如何“间接减少渲染”？

它的作用是：

配合 React.memo / PureComponent，避免子组件因函数 props 变化而重复渲染

例子：

const Child = React.memo(({ onClick }) => {
  console.log("child render");
  return <button onClick={onClick}>click</button>;
});

const Parent = () => {
  const handleClick = useCallback(() => {
    console.log("click");
  }, []);

  return <Child onClick={handleClick} />;
};

对比：

❌ 不用 useCallback

  • Parent re-render → handleClick 重新创建
  • Child props 变化 → Child re-render

✅ 用 useCallback

  • Parent re-render
  • handleClick 引用不变
  • Child props 没变 → Child 不 re-render

什么时候需要用 useCallback

useCallback 的使用前提是“函数的引用变化会导致额外的渲染或副作用触发”，典型场景包括 React.memo 子组件的 props 稳定性、useEffect 依赖稳定性，以及高频渲染场景下避免函数引用变化导致的渲染扩散。否则它只是增加代码复杂度，没有实际收益。

①作为 React.memo 子组件的 props（最核心）

②作为 useEffect / useMemo 依赖（防止 effect 重跑）

const fetchData = useCallback(() => {
  api.getData();
}, [id]);

useEffect(() => {
  fetchData();
}, [fetchData]);

如果不用 useCallback：

fetchData 每次 render 都变
→ effect 每次都触发
→ 变成“无限重复逻辑风险”

👉 本质：稳定依赖

③高频场景 + 大量子组件依赖（性能扩散控制）

  • 列表（100～1000+ items）
items.map((item) => <Row onClick={handleClick} />);
  • WebSocket / 实时数据 UI
  • drag / scroll / animation 交互
  • 大型复杂组件树（dashboard / editor）

👉 关键不是“函数创建”，而是：
避免一个函数变化 → 导致 N 个子组件连锁 re-render

Q1：useCallback 能减少自己组件的 render 吗？

❌ 不能
它不控制 render，只控制“函数引用是否变化”

Q2：什么时候用 useCallback 没意义？

  • 子组件没用 React.memo。
    如果子组件没用 React.memo，就等于：👉 你即使把函数缓存住了，子组件也还是会照常 re-render
  • 函数没作为 props 传下去
  • 渲染成本本身很低

👉 用了反而增加复杂度

3. useCallback 依赖项是深比较还是浅比较？

useCallback 依赖项 = 逐个 Object.is 的浅比较(shallow comparison)，不做深比较，也不追踪对象内部变化。

内部使用的是 Object.is 算法进行严格相等比较。如果依赖项是对象或数组，它只比较内存引用是否改变，不会递归比较对象内部属性。

基本类型

useCallback(fn, [1, 2]);

1 === 1 ✔

2 === 2 ✔

👉 不触发更新

❗ 引用类型（重点）

const obj = { a: 1 };
useCallback(fn, [obj]);

即使：

obj.a = 2;

👉 只要引用没变：

❌ 不会触发更新

但如果：

setObj({ a: 2 });

👉 新对象引用：

✔ 触发更新

警告：如果依赖项是 { a: 1 }，即使 a 没变，但父组件每次渲染都新建这个对象，useCallback 依然会认为依赖变了，导致缓存失效。所以依赖项最好传基本类型（string, number, boolean）或useState返回的稳定setter。

为什么不是深比较？

如果是深比较，每次 render 都需要递归遍历对象结构，会显著增加 render 阶段的计算开销。浅比较是为了避免深比较带来的性能开销，同时也符合 React 的设计理念——依赖变化由引用变化来表达，因此对象或数组依赖需要开发者自行保证引用稳定性。

一个经典坑（面试高频）

❌ 错误写法

useCallback(fn, [{ a: 1 }]);

你以为：

👉 a 变了会触发

但实际上：每次 render 都是新对象，每次都触发 callback 更新

✔ 正确写法

const { a } = obj;

useCallback(fn, [a]);

4. useCallback 和 useMemo 比较的是什么？（本质区别）

useCallback 和 useMemo 的依赖项比较方式是相同的，都是通过 Object.is 对依赖数组逐项进行浅比较。它们并不比较函数或返回值本身，而是通过依赖是否变化来决定是否重新计算或重新创建函数。其中 useMemo 缓存的是计算结果，而 useCallback 缓存的是函数引用。

Hook 缓存的是什么
useMemo “计算结果（value）”
useCallback “函数（function）”

useCallback(fn, deps) 在实现上等价于 useMemo(() => fn, deps)，本质都是基于依赖数组的浅比较决定是否返回缓存值。但两者在语义上不同：useMemo 用于缓存计算结果，而 useCallback 用于缓存函数引用，以提高代码可读性和表达意图。

一句话总结：useCallback 保证函数地址不变；useMemo 保证计算出的值不变（跳过重复计算）。
