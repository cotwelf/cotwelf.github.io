前端 基础 JavaScript

1. Event Loop 的执行顺序

  JS 是单线程语言，同一时间只能做一件事，不能有多个线程。所有在主线程上的任务都需要排队依次执行。可以把 JS 想象成在时间线上运行的。

Event Loop (事件循环)，是指浏览器或 Node 环境的一种解决 JS 单线程运行时不会阻塞的一种机制，也就是异步的原理，作为单线程语言，JS 本身没有异步这一说法，是其宿主环境提供的。

  • 解释器判断一行 JS 代码是同步还是异步，同步则进入 主线程，异步进入 event table;
  • 进入主线程的同步代码，优先执行 宏任务，微任务进入 任务队列，主线程中的异步代码推入 event table；
  • 主线程中同步任务执行结束，去 event queue 中查看是否有可执行的异步任务，有则推入主线程，重复上述步骤，直到主线程和 event table 清空。

每一次/层循环，都是首先从宏任务开始，微任务结束。每一个宏任务执行完之后，都会检查是否存在待执行的微任务，如果有，则执行完所有微任务之后，再继续执行下一个宏任务。

【名词解释】

  • （1）进程和线程

      浏览器每一个 tab 都可以看做是浏览器内核进程，每个进程下面会有多个线程来互相配合完成任务。比如 GUI 线程、JS 引擎线程、网络线程、定时器线程等。

  • （2）任务队列

      任务队列中存放着微任务，会在主线程内任务全部执行完毕之后读取并执行。

  • （3）宏任务和微任务

      JS 的任务可以分为两种：

    • 微任务（MircoTask）：耗时任务
      • Promise中的then、catch、finally 原理可戳 【js进阶】手撕Promise，一码一解析 包懂！
      • MutationObserver 等
    • 宏任务（MacroTask/Task）：非耗时任务
      • 基本上，我们将javascript中非微任务（MircoTask）的所有任务都归为宏任务

  • （4）同步和异步

    • 同步任务：JS 执行时就在主线程上排队执行的任务
      • 定时器 setTimeout 和 setInterval：设置定时器是同步（立即设置），异步指的是间隔多久后执行指定的函数
      • 事件绑定（监听）
      • AJAX 的异步请求（fetch 应该也是
      • promise/async/await
    • 异步任务：会进入 event table 等候进入主线程任务结束之后进入的任务

  • （5）执行栈
      执行栈是解释器追踪函数执行流的一种机制。当引擎第一次遇到 js 代码时，会产生一个全局执行上下文压入执行栈，每遇到一个函数调用，就往栈中压入一个新的 上下文。引擎执行栈顶的函数后则弹出当前执行上下文直至函数依次执行完毕并回到全局的上下文。

举个栗子：

console.log('1');

setTimeout(function() { // T1
  console.log('2');
  new Promise(function(resolve) { // P1
      console.log('3');
      resolve();
  }).then(function() {
      console.log('4')
  })
  setTimeout(function() { // T2
    console.log('5');
    new Promise(function(resolve) { // P2
        console.log('6');
        resolve();
    }).then(function() {
        console.log('7')
    })
  })
  console.log('14');
})

new Promise(function(resolve) { // P3
    console.log('8');
    resolve();
}).then(function() {
    console.log('9')
})

setTimeout(function() { // T3
  console.log('10');
  new Promise(function(resolve) { // P4
      console.log('11');
      resolve();
  }).then(function() {
      console.log('12')
  })
})
console.log('13')
// 1, 8, 13, 9, 2, 3, 14, 4, 10, 11, 12, 5, 6, 7

分析一下上面的过程（setTimeout: T, Promise: P, console.log: L)

JS 开始执行，同步任务 [L1, P3, L13] 进入主线程，异步任务 [T1, T3] 进入 event table 等待执行

  • [L1, P3, L13] 在主线程
    • 宏任务 [L1, P3-L8, L13]
    • 微任务 [P3.then-L9]
    • event table [T1, T3]

      打印出 1, 8, 13, 9，主线程清空，准备处理 event table 中的 T1

  • eventTabel[0] 即 setTimeout T1 进入主线程
    • 宏任务 [L2, P1-L3, L14]
    • 微任务 [P1.then-L4]
    • event table [T3, T2]

      打印出 2, 3, 14, 4，主线程清空，准备处理 event table 中的 T3

  • eventTabel[0] 即 setTimeout T3 进入主线程
    • 宏任务 [L10, P4-L11]
    • 微任务 [P4.then-L12]
    • event table [T2]

      打印出 10, 11, 12，主线程清空，准备处理 event table 中的 T2

  • eventTabel[0] 即 setTimeout T2 进入主线程
    • 宏任务 [L5, P2-L6]
    • 微任务 [P2.then-L7]
    • event table []

      打印出 5, 6, 7，主线程清空，event table 清空

宏任务和微任务是相对而言的，根据代码执时循环的先后，将代码执行分层理解，在每一层（一次）的事件循环中，首先整体代码块看作一个宏任务，宏任务中的 Promise（then、catch、finally）、MutationObserver、Process.nextTick就是该宏任务层的微任务；宏任务中的同步代码进入主线程中立即执行的，宏任务中的非微任务异步执行代码将作为下一次循环的宏任务时进入调用栈等待执行的；此时，调用栈中等待执行的队列分为两种，优先级较高先执行的本层循环微任务队列（MicroTask Queue），和优先级低的下层循环执行的宏任务队列（MacroTask Queue）！

定时器对队列的工作方式是，当特定时间过去后将代码插入。注意，给队列添加代码并不意味着对它立刻执行，而只能表示它会尽快执行。设定一个 150ms 后执行的定时器不代表到了 150ms 代码就立刻执行，它表示代码会在 150ms 后被加入到队列中。如果在这个时间点上，队列中没有其他东西，那么这段代码就会被执行，表面上看上去好像代码就在精确指定的时间点上执行了。其他情况下，代码可能明显地等待更长时间才执行。
