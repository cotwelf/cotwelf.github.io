记一次非常非常非常麻烦的浏览器兼容问题

前端 兼容 响应式

先说总结

检查设计稿检查设计稿检查设计稿！！！（重要的事情说三遍！）

如果设计说，【一屏显示，不允许滑动。】，那么拿到图之后：

step1：看 web 设计稿尺寸是否是 1920 * 1080。是：转到 step2，否：计算下宽度，宽高比最好是 1.78 - 1.96 左右（1920 * 978），否则可能要考虑字体大小的问题；

step2：检查页面上最小字体是否小于等于 14px。（小于等于 12px 可以怼更狠一点）是：转到 step4，否：转到 step3；

step3：确认需求方是否要求适配最小屏幕（即显示器分辨率 1366 * 768）。如果需要兼容，可能需要修改字体大小。什么？你不知道改成多少？？你 tm 不是专业设计么？！

step4：设计稿打回去，并大声问设计：你的电脑没有系统菜单栏吗？？？？？你的浏览器没有导航甚至收藏夹吗？？？？？？重做！重做！！重！！做！！！

故事的开始

一切痛苦的根源，是我最近接了个需求，设计说一屏展示，不允许滑动。设计稿尺寸 1920 * 1080，看了下页面最小字体是 14px。

聪明宝宝请在这个节点就打回设计稿，并狂怼设计！！

鸡汁的我想到了 rem。好么，我都用 rem，用户进来的时候先判断视口宽高，只要有宽小于 1920 或高小于 1080，我就等比缩放根节点的 font-size，优秀！不愧是我！！

在做的过程中其实有意识到，为什么是严格按照设计稿比例转的 rem，为什么最小字体还是要比设计稿上的小呢？当时的我并没有很在意，以为是我屏幕分辨率不够优秀叭 orz（现在想想我真是个傻子qwq）

其实，能走到这一步，说明我已经不那么聪明了0.0

PR 通过 review（毕竟开发不会用超小屏为难自己人，不然看代码眼要瞎掉，所以 review 还是通过了的），提测之后，产品拿着图来找我，诶，大佬，我这里不对啊，文字出框了 qwq。。。为啥你那里效果这么好啊。。。

当时我。。。

好么，那我给你放大点还不行么。于是相当于缩小了页面上下边距。但。。。。还是有问题。。。我就开始怀疑字体是不是已经超过 chrome 预设的 12px，啪，果然。

到这里，这只傻狍子还是没发现设计稿的问题，并坚信是自己适配做的不好（啪啪！）

于是傻狍子说服设计改成大屏一屏，小屏可滑动，随后更改了 js，比较根节点缩放后的 font-size 下，最小的那个 14px 会不会小于 12px，如果小了，那就缩放到 12px 停止；同时在 scss 里给每个容器设置了最小宽高。

后来和大佬吐槽，大佬问，你们产品拿的多大的电脑看的？我说 1366 * 786，大佬直接笑了，说他们产品给他提 786 的 BUG 一概不修，现在哪里还有 786 啊哈哈哈。

直到今天。。。我看到群里设计和产品在说，需求方吐槽我没有精准实现设计稿，玻璃心的我抱着学习的心态，再次想看看自己的代码有什么需要改进的，才发现这个 BUG 的源头在设计稿。。。于是有了这篇文档

好么，小丑竟是我寄几 QAQQAQQAQ

如果再遇到一屏展示，需要设计稿的同时，还要字体包（因为有些设备字体不同，不同字体可能会出现同 size 但大小不一的状况），不要用 rem 了，按设计稿来，多少 px 就多少 px，js 判断视口宽高 scale 缩放（字体小就小吧，设计稿就这样的（叉腰））

肥肠抱歉！！这是一个只有主银才能看的文档哦 _(:з」∠)_
